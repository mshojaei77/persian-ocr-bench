## ۱. هدف پروژه

ساخت و انتشار یک **سیستم OCR متن‌باز و دقیق** برای استخراج متن از اسناد فارسی **چاپی و دست‌نویس**، همراه با:

- یک **دیتاست استاندارد** (واقعی + مصنوعی) با لیبل مرجع باکیفیت
- یک **بنچمارک قابل‌بازتولید** روی ده‌ها مدل OCR و VLM متن‌باز
- یک **لیدربورد عمومی** با خروجی خام مدل‌ها برای بررسی مستقل
- **مدل‌های فاین‌تیون‌شده** منتشرشده در Hugging Face و کد در GitHub

خروجی نهایی باید روی موارد فidelity‑محور (اسناد حقوقی، مالی، اداری) قابل اتکا باشد، نه فقط متنی که «شبیه فارسی» به نظر برسد.

## ۳. فازهای پروژه (۷ فاز)

### فاز ۱ — جمع‌آوری داده و انتخاب مدل‌ها
**وضعیت: در حال انجام**
- فراخوان عمومی برای ارسال اسکن/عکس اسناد فارسی (چاپی و دست‌نویس)
- لیبل‌گذاری با مدل معلم بزرگ + بازبینی انسانی
- **بنچمارک ۵۰+ مدل** OCR و Vision‑Language متن‌باز
- **انتخاب ۳ تا ۵ مدل برتر** بر اساس دقت، نوع معماری و اندازه مدل
- جمع‌آوری و کراول داده از وب، Hugging Face، Kaggle، منابع دانشگاهی و آرشیوها
- پاک‌سازی، ناشناس‌سازی (PII) و بررسی مجوز انتشار داده‌ها
- **خروجی فاز ۱:** فایل نتایج CSV/JSON، لیدربورد اولیه، خروجی خام مدل‌ها، گزارش خطا، نوت‌بوک‌های Colab

### فاز ۲ — جمع‌آوری و آماده‌سازی داده واقعی
- تفکیک داده‌ها بر اساس: چاپی/دست‌نویس، نوع سند، کیفیت تصویر، فونت/سبک، پیچیدگی صفحه
- حذف یا محو اطلاعات شخصی و محرمانه
- تولید Ground Truth با مدل معلم + تایید انسانی
- کاتالوگ‌سازی منابع (نگاه کنید به بخش ۷)

### فاز ۳ — تولید داده مصنوعی (Synthetic)
- تولید **~۱۰۰ هزار تا ۱ میلیون نمونه** با فونت‌های متنوع فارسی
- پشتیبانی از: متن تک‌خطی، پاراگراف، جدول، فرم، سند چندستونه، متن فارسی–انگلیسی
- افزودن نویز واقعی: تاری، چرخش، پرسپکتیو، سایه، فشرده‌سازی JPEG، پس‌زمینه کاغذ، کیفیت پایین اسکن
- **انتشار یک رندرر متن‌باز** برای تولید داده مصنوعی فارسی

### فاز ۴ — فاین‌تیون و بهینه‌سازی
- فاین‌تیون ۳ تا ۵ مدل منتخب (ترجیحاً **LoRA / QLoRA** برای جلوگیری از فراموشیِ دانش پیش‌آموخته)
- آموزش جداگانه یا ترکیبی برای: متن چاپی، دست‌نویس، اسناد پیچیده
- آزمایش روش‌های پیش‌پردازش تصویر
- توسعه **پست‌پردازش اختصاصی فارسی** (نرمال‌سازی ی/ي، ک/ك، نیم‌فاصله، اعداد)
- مقایسه مدل پایه در برابر نسخه فاین‌تیون‌شده + ثبت کامل تنظیمات

### فاز ۵ — ارزیابی نهایی و لیدربورد عمومی
- ایجاد **مجموعه تست استاندارد و ثابت** (جلوگیری از نشت تصاویر تست به آموزش)
- ارزیابی به تفکیک دسته‌های سند
- انتشار لیدربورد شامل: CER/WER کلی، دقت چاپی، دقت دست‌نویس، Exact Match، عملکرد به تفکیک نوع سند
- انتشار خروجی خام + اسکریپت ارزیابی قابل‌بازتولید

### فاز ۶ — انتشار
- انتشار مدل‌ها در Hugging Face + کد در GitHub
- دموی تحت وب، نوت‌بوک Colab، مستندات نصب/استفاده، API + مثال
- Model Card و Dataset Card، اعلام مجوز، محدودیت‌ها و نتایج ارزیابی

### فاز ۷ — توسعه جامعه
- نگهداری مدل‌ها و دیتاست، پذیرش PR، ثبت Issue/Discussion
- برگزاری هکاتون و چالش OCR فارسی
- گسترش به: اسناد تاریخی، نسخه‌های خطی، جدول/فرم، تشخیص ساختار صفحه، و زبان‌های هم‌خط (کردی، اردو، پشتو)

## ۴. فرمت داده و لیبل (Data & Label Format)

پروژه دو سطح فرمت دارد:

### الف) فرمت حداقل — برای مشارکت‌های جامعه (تاپیک Data: Already Labeled)
- ارسال به صورت **ZIP یا RAR**
- فایل اصلی (تصویر/سند/PDF) و فایل لیبل باید **دقیقاً هم‌نام** باشند:
  - `data_1.png` + `data_1.txt` (یا `data_1.md`)
  - `data_2.pdf` + `data_2.md`
- لیبل **فقط** در قالب `.txt` یا `.md` (کدگذاری ترجیحاً UTF‑8)
- هر صفحه جداگانه نام‌گذاری شود (`book_001.png` + `book_001.txt`)
- لیبل باید **دقیقاً مطابق محتوای سند** باشد: بدون ترجمه، اصلاح املا یا توضیح اضافی
- جزئیات کامل در `docs/telegram/TOPIC_PINS.md` (تاپیک Data: Already Labeled)

### ب) فرمت ساختاریافته — برای پایپ‌لاین داخلی / هم‌راستا با Model Card
بر اساس الگویی که در گفتگو از Model Card یکی از مدل‌ها استخراج شد:

| فیلد | توضیح |
|------|-------|
| `image` | عکس خام سند |
| `image_with_layout` | همان عکس با کادرهای تشخیص (visualization) — از dots.ocr |
| `raw_text` | متن خام خط‌به‌خط |
| `markdown` | متن ساختاریافته: heading + لیست |
| `tables_html` | جدول‌ها به HTML — از OmniDocBench |
| `formulas_latex` | فرمول‌ها به LaTeX — از OmniDocBench |
| `blocks_json` | `[{category, bbox:[x1,y1,x2,y2], text, confidence}]` — از Surya/Falcon |
| `reading_order` | ترتیب صحیح خواندن بلاک‌ها |
| `doc_type` | نوع سند (مقاله/فاکتور/فرم/کاتالوگ) |
| `language` | fa / en / mixed |

### قوانین لیبل‌گذاری فارسی (نکات کلیدی از گفتگو)
- **متن را verbatim بنویسید**؛ مدل معلم گاهی اصلاح یا ترجمه می‌کند — انسان باید تصحیح کند.
- **اختلاف ی/ي/ى و ک/ك** را حفظ کنید؛ در لایه «Strict CER» این‌ها متمایزند.
- **نیم‌فاصله (ZWNJ)** بخشی از املای فارسی است، نه تزیین — آن را پاک نکنید مگر در لایه نرمال‌شده.
- **اعداد:** فارسی (`۱۲۳`)، عربی‑اندیک (`١٢٣`) و لاتین (`123`) متفاوت‌اند؛ در لیبل دقیقاً همان‌چیزی که در سند هست را بنویسید.
- **RTL/LTR مخلوط:** فارسی نه کاملاً راست‌به‌چپ و نه چپ‌به‌راست است. هنگام بازبینی لیبل در ویرایشگر، از راست‌چین صحیح استفاده کنید (نوت‌پد ویندوز گاهی عدد لاتین را فارسی نشان می‌دهد — VS Code دقیق‌تر است اما چپ‌به‌راست نمایش می‌دهد).
- **ترتیب خواندن** در اسناد چندستونه، جداول و فرمول‌ها چالش‌برانگیز است؛ در سطح ساختاریافته (`reading_order`) ثبت شود.

## ۵. معیارهای ارزیابی (Metrics)

جزئیات کامل در `docs/benchmark_metrics.md`. خلاصه:

- **معیارهای اصلی:** CER (نرخ خطای کاراکتر)، WER (نرخ خطای کلمه)، Exact Match، Normalized Edit Distance
- **معیارهای اختصاصی فارسی:** تفاوت ی/ي، ک/ك، نیم‌فاصله، اعداد فارسی/انگلیسی، علائم نگارشی، متن ترکیبی RTL/LTR، دقت ترتیب خواندن
- **سه لیدربورد پیشنهادی:**
  1. **Persian Text Recognition** — تشخیص متن (خط/بلوک تکی)
  2. **Persian Document Parsing** — فقط مدل‌های صفحه‌محور (reading order، layout، جدول، فرمول)
  3. **Specialized Persian OCR** — دست‌خط، متن صحنه‌ای، ارقام
- **قابلیت اطمینان VLM:** نرخ حذف/درج/جایگزینی، نرخ شکست فاجعه‌بار (CER>۰.۵)، خروجی خالی/نامعتبر
- **مقاومت (Robustness):** نگه‌داشت نمره روی نسخه‌های تخریب‌یافته (بلur، JPEG، چرخش، سایه، کاغذ رنگی...)

**نقش‌های پایپ‌لاین (طبق گفتگو):**
| نقش | مدل | توضیح |
|------|------|-------|
| Teacher (معلم) | Gemini 3.1 Flash Lite | تولید لیبل + تعیین نوع/سختی سند |

**مدل‌های VLM/متن‌باز قوی برای بنچمارک:**
- PaddleOCR-VL-1.6 (~۰.۹B، ۸۹٪)، PaddleOCR-VL-0.9B (۸۷٪)

- DeepSeek-OCR-2 (۳B، ۵۸٪ — پشتیبانی خوب عربی/فارسی ولی نیاز به ارزیابی دقیق)

**منابع تجاری (برای مقایسه/بازبینی، نه انتشار):**
- Mistral OCR (۹۰٪)، Google Document AI (۹۱٪)، Azure AI Document Intelligence (۹۲٪)، Nanonets OCR-3 (۹۰٪)

## ۸. چالش‌های شناسایی‌شده (Key Challenges)

1. **لیبل‌گذاری سخت‌ترین بخش است** — حتی مدل‌های بزرگ (Gemini/GPT) اشتباه می‌کنند؛ ~۲۰٪ خروجی‌های معلم نیاز به اصلاح دارند.
2. **RTL/LTR مخلوط + اعداد** — ترتیب خواندن و تمایز اعداد فارسی/عربی/لاتین، مدل را دچار سردرگمی می‌کند.
3. **هالوسیناسیون VLM** — خروجی «روان اما نادرست» برای متن حقوقی/مالی خطرناک است؛ در جاهایی که fidelity اهمیت دارد، یک مرحله تشخیص deterministic نگه دارید.
4. **شکاف دامنه مصنوعی/واقعی** — داده مصنوعی با اسکن واقعی فاصله دارد؛ همیشه بخشی از صفحات واقعی را برای ارزیابی نهایی کنار بگذارید.
5. **چیدمان برای اسناد پیچیده** — جداول، فرم‌ها و چندستونه‌ها نیازمند layout detection (Surya/Granite/dots) هستند.
6. **مجوز و حریم خصوصی** — حذف PII و تایید اجازه انتشار پیش از افزودن هر داده به دیتاست عمومی.
